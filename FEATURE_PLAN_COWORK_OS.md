# แผนฟีเจอร์: เอาแนวคิด "Cowork OS" มาใส่ LM Co-work — 2 ก.ค. 2026

เทียบ 8 ขั้นตอนของแนวคิด Claude Cowork OS กับสิ่งที่ LM Co-work **มีอยู่แล้วจริงในโค้ด**
เพื่อหาว่าอะไรมีแล้ว อะไรต่อยอดนิดเดียวได้ และอะไรต้องสร้างใหม่

## ตารางเทียบ: แนวคิด vs สิ่งที่มีอยู่

| ขั้นตอนใน Cowork OS | สถานะใน LM Co-work | หมายเหตุ |
|---|---|---|
| 1–2. โฟลเดอร์งานเป็นพื้นที่ทำงาน AI | ✅ มีครบ | workspace + path jail + โหมด Co-Work |
| 3. ไฟล์คำสั่งหลัก + ไฟล์ความจำ | ❌ ยังไม่มี | มี knowledge_store (RAG) แต่ไม่ใช่ memory ที่ inject อัตโนมัติ → **F1** |
| 4. Connectors (Gmail/Calendar/Drive) | 🟡 มีโครงแล้ว | mcp_client.py รองรับ mcp.json แต่ไม่มี UI + ไม่มี confirm gate (SEC-6) → **F4** |
| 5. Workflow งานประจำ | 🟡 ทำได้ผ่าน prompt skill | แต่ผู้ใช้ต้องรู้เอง ไม่มีตัวอย่าง/เทมเพลต → **F6** |
| 6. ทดสอบ workflow | ✅ | เป็นเรื่องการใช้งาน ไม่ใช่ฟีเจอร์ |
| 7. บันทึก workflow เป็น Skill | ✅ มี UI สร้าง skill แล้ว | ต่อยอด: ปุ่ม "บันทึกแชตนี้เป็น Skill" → **F2** |
| 8. Scheduled tasks / สั่งงานมือถือ | ❌ ยังไม่มี | มี system tray + hotkey แล้ว เป็นฐานที่ดี → **F5** (มือถือ = นอกขอบเขต แอป local) |

## ฟีเจอร์ที่เสนอ (เรียงตามความคุ้ม: ผลลัพธ์/แรงที่ลง)

### F1 🥇 Memory file — ไฟล์ความจำที่ inject อัตโนมัติ (เล็ก, ~ครึ่งวัน)
หัวใจของ "Second Brain" ทั้งโพสต์ และเป็นชิ้นที่ขาดอยู่จริง:

- ถ้าในโฟลเดอร์งานมี `_agent.md` (คำสั่งประจำโฟลเดอร์) และ `_memory.md` (สิ่งที่ให้จำ)
  → `run_agent()` อ่านแล้วต่อท้าย system prompt อัตโนมัติ (จุดเดียวกับที่ต่อ
  `COWORK_PROMPT`/`skills_catalog` อยู่แล้วใน server.py)
- เพิ่ม tool `remember(text)` ลง `tools.py`: append ลง `_memory.md` — เข้า
  `WRITE_TOOLS` ให้ผ่าน confirm flow เดิมโดยไม่ต้องเขียนอะไรใหม่
- **ข้อจำกัดที่ต้องออกแบบเผื่อ:** โมเดล local context สั้น (4k–8k) → cap ไฟล์ความจำ
  ที่ ~3,000 ตัวอักษร ถ้าเกินให้ตัดหัว (ความจำเก่าสุดหายก่อน) + แสดงขนาดใน UI

### F2 🥈 ปุ่ม "บันทึกแชตนี้เป็น Skill" (เล็ก, ~2–3 ชม.)
ปิด loop ขั้นตอนที่ 7 ให้เหลือคลิกเดียว: ปุ่มในแชต → เอาข้อความ user คนแรก +
คำตอบสุดท้ายไปเป็นร่าง prompt skill → เปิดฟอร์ม create-skill (มีอยู่แล้ว) แบบ
กรอกให้ล่วงหน้า ผู้ใช้แค่ตั้งชื่อแล้วกดบันทึก — backend ไม่ต้องแตะเลย
(`SL.create_skill` รองรับ prompt skill อยู่แล้ว)

### F3 🥉 ปุ่ม "จัดโครงสร้างโฟลเดอร์เริ่มต้น" (เล็ก, ~2 ชม.)
ขั้นตอนที่ 3 ของโพสต์แบบสำเร็จรูป: ปุ่มใน UI → สร้าง `_agent.md`, `_memory.md`,
`inbox/`, `projects/`, `archive/` + คำอธิบายสั้นในแต่ละไฟล์ ผ่าน proposal-confirm
flow เดิม (ผู้ใช้เห็นก่อนว่าจะสร้างอะไรบ้าง) — ใช้คู่กับ F1

### F4 MCP Connectors UI + confirm gate (กลาง, ~1–2 วัน)
ยกระดับขั้นตอนที่ 4 จาก "แก้ไฟล์ mcp.json เอง" เป็นจัดการใน UI:

- แท็บ Connectors: เพิ่ม/ลบ MCP server (command + args + env), สถานะ
  ต่อติด/ไม่ติด, รายชื่อ tools ที่ได้มา
- **ต้องทำพร้อมกัน:** ปิด SEC-6 จาก audit — MCP tool ครั้งแรกต้องผ่าน confirm
  เหมือน skill (โค้ด `_handle_tool_call` มี hook อยู่แล้ว เพิ่มเงื่อนไข
  `fname in mcp_manager.tool_mapping` บรรทัดเดียว)
- แก้ QUAL-5 (stderr deadlock + load ตอน import) ไปด้วยเลยเพราะแตะไฟล์เดียวกัน
- หมายเหตุ: Gmail/Calendar MCP ส่วนใหญ่ต้องใช้ npx/node — ต้องเขียนบอกผู้ใช้ชัด

### F5 Scheduled tasks (กลาง–ใหญ่, ~2–3 วัน)
ขั้นตอนที่ 8 — ทำได้เพราะมี tray icon + background thread อยู่แล้ว:

- `data/schedules.json`: [{id, cron/เวลา, agent, prompt, workspace, enabled}]
- Scheduler thread เช็คทุกนาที → ครบเวลา → เรียก `run_agent()` แบบ headless
  → เขียนผลลง `workspace/reports/YYYY-MM-DD-<ชื่องาน>.md` + toast/notification
- UI: แท็บเล็ก ๆ ตั้งเวลา เลือก agent + prompt
- **ต้องทำ OBS-1 (log file) ก่อน** ไม่งั้นงานที่รันตอนไม่มีใครดูจะพังเงียบ
  แบบไม่มีร่องรอยเลย
- ข้อจำกัดที่ต้องบอกผู้ใช้ตรง ๆ: ทำงานเฉพาะตอนแอปเปิดอยู่ (ไม่ใช่ Windows service)

### F6 คลังเทมเพลต Workflow (เล็ก, ~ครึ่งวัน)
ขั้นตอนที่ 5 แบบไม่ต้องเขียนโค้ด: ชุด prompt skill สำเร็จรูป 4–5 ตัวติดมากับแอป
เช่น `สรุปโฟลเดอร์ประจำวัน`, `ร่างใบเสนอราคา`, `วางแผนคอนเทนต์รายสัปดาห์`,
`คัดกรองไฟล์ inbox` — เป็นแค่โฟลเดอร์ใน `skills/` เพิ่ม (ระบบโหลดให้เองอยู่แล้ว)
และเป็นตัวอย่างสอนผู้ใช้สร้างของตัวเองไปในตัว

## ลำดับที่แนะนำ

```
รอบ 1 (วันเดียวจบ, เห็นผลทันที):   F1 → F3 → F6
รอบ 2 (ปิดหนี้เทคนิคไปด้วย):       OBS-1 → F4 (พ่วงแก้ SEC-6 + QUAL-5)
รอบ 3 (ฟีเจอร์เรือธง):             F5 → F2
```

เหตุผล: รอบ 1 คือแก่นของ "Second Brain" ที่โพสต์พูดถึง ใช้โครงที่มีอยู่เกือบ 100%
รอบ 2 ทำ connectors ให้ปลอดภัยก่อนเปิดใช้จริง รอบ 3 คือ automation เต็มรูปแบบ
